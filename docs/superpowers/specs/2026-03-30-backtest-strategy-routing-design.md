# 单次回测策略路由与契约分层设计

## 1. 文档目的

本文用于收敛“单次回测先做成，同时兼容后续策略扩展”的接口设计结论，重点回答以下问题：

1. 当前单次回测接口应该按什么方式组织。
2. 为什么当前阶段不选择单入口多策略分发。
3. 请求体、响应体、服务层边界应该如何划分，才能避免后续出现兼容代码、双轨字段和过度设计。

本文只讨论接口设计与边界约定，不进入真实回测引擎实现。

## 2. 当前约束与判断前提

当前项目处于 MVP 初期，目标是先完成单个策略的单次回测闭环，而不是提前做成完整策略平台。

本次设计必须同时满足以下约束：

1. 不允许兼容代码、兜底代码、双字段并存。
2. 不允许为了未来假设需求提前引入过重抽象。
3. 需要给后续新增策略保留清晰扩展位，但不能因此污染当前 DCA 契约。
4. 前后端最终应收敛到一套清晰、稳定、可生成类型的真实契约。
5. 当前项目更重视可读性、可解释性和学习成本，而不是平台化程度。

## 3. 主流方案概览

### 3.1 方案一：单入口 + 判别联合

典型形式：

- `POST /api/v1/backtests/run`
- 请求体包含 `strategy_id`
- 按 `strategy_id` 映射到不同策略请求模型
- OpenAPI 中一般表达为 `oneOf + discriminator`

这是平台型、多策略统一调度系统中较主流的方案。优点是入口统一，后续扩展在接口表面上更规整。缺点是当前阶段会引入更高的理解成本，尤其是对于学习型项目，前后端都需要更早接受判别联合、类型缩小和策略分发的概念。

### 3.2 方案二：每个策略独立路由

典型形式：

- `POST /api/v1/backtests/dca/run`
- `POST /api/v1/backtests/grid/run`
- `POST /api/v1/backtests/value-average/run`

这是业务明确、策略数量有限、优先追求简单和落地速度的项目里非常常见的工程方案。优点是请求含义直观，OpenAPI 文档清晰，用户和开发者都更容易理解。缺点是平台化程度不如单入口统一分发，但这并不是当前项目的主要目标。

### 3.3 方案三：单入口 + `strategy_params`

典型形式：

- `POST /api/v1/backtests/run`
- 公共字段放外层
- 策略私有字段都塞进 `strategy_params`

这类方案实现快，但类型会迅速变松，文档和调试体验都较差。对于当前明确强调“单一口径、禁止兼容、避免大口袋字段”的项目，不适合采用。

## 4. 最终设计结论

本项目当前阶段采用方案二：

1. 每个策略使用自己的 `run` 路由。
2. 请求体采用 `Base + 策略子请求`。
3. 响应体采用 `统一外壳 + 策略子响应`。
4. 请求体不再保留 `strategy_id`。
5. 响应体保留 `strategy_id` 作为结果元信息。

这一定义的核心原因不是“喜欢按路由拆”，而是要保持唯一事实来源：

1. 路由负责表达“执行哪个策略”。
2. 请求体负责表达“这个策略运行所需的参数”。
3. 响应体负责表达“本次运行产出的通用结果和结果元信息”。

如果已经按策略拆路由，就不应再让请求体继续携带 `strategy_id`，否则会出现双重真相源，后续容易逼出兼容判断和无意义校验。

## 5. 请求体分层约定

### 5.1 Base 请求结构

`BacktestRunRequestBase` 只允许放“所有策略都稳定共享的请求语义”。

当前可归入 Base 的字段如下：

1. `symbol`
2. `start_date`
3. `end_date`
4. `initial_capital`
5. `cost_model`

这些字段属于当前明确认定的跨策略共享基础输入，而不是“看起来暂时通用”的字段。

### 5.2 DCA 请求结构

`DcaBacktestRunRequest` 继承 `BacktestRunRequestBase`，只承载 DCA 自己的输入字段，例如：

1. `periodic_investment_amount`
2. `investment_frequency`
3. `monthly_investment_day`
4. `weekly_investment_weekday`

这些字段的语义明显依赖“定投策略”本身，因此不能进入 Base。

### 5.3 Base 的硬边界

只要某个字段不是“所有策略都稳定共享”，就不能进入 Base。这里需要特别防止两类错误：

1. 因为当前只有 DCA，所以误把 DCA 字段当成公共字段。
2. 因为未来可能有相似策略，所以提前把策略私有字段抽进公共层。

Base 不是“当前觉得常用”的字段集合，而是“跨策略仍然稳定成立”的字段集合。

## 6. 响应体分层约定

### 6.1 统一响应外壳

所有策略共享统一响应外壳，用于保证结果页、导出层和后续 AI 报告输入层的基础稳定性。

当前可放入 `BacktestRunResponseBase` 的字段如下：

1. `run_id`
2. `request_id`
3. `strategy_id`
4. `engine_version`
5. `metric_definition_version`
6. `summary`
7. `nav_series`
8. `drawdown_series`
9. `capital_curve`
10. `records`
11. `message`

这些字段表达的是运行追踪信息与通用结果视图，不依赖某个特定策略的输入细节。

### 6.2 策略子响应

`DcaBacktestRunResponse` 继承 `BacktestRunResponseBase`，并声明该策略自己的结果细节。

当前已确认的策略子响应差异点：

1. `input_snapshot` 放在策略子响应中，而不是 Base 中。

原因如下：

1. `input_snapshot` 的类型天然跟随策略请求体变化。
2. 如果把 `input_snapshot` 放进 Base，会迫使公共层承担联合类型或泛型复杂度。
3. 当前项目阶段不需要为了一个字段把整个响应体系泛型化。

### 6.3 `strategy_result` 的处理

设计上允许未来出现 `strategy_result` 这样的策略专属结果区块，但当前阶段不提前添加空字段。

也就是说：

1. 当前没有真实策略专属结果，就不创建 `strategy_result: null` 或空对象。
2. 等某个策略真实需要额外结果区块时，再定义明确 schema 并加入对应子响应。

这样做可以避免为了未来假设需求制造空扩展位，符合反过度设计原则。

## 7. 路由层与服务层约定

当前阶段采用最小实现：

1. `dca route -> dca service`
2. 路由层只做参数接收、调用服务、返回 schema
3. 真实业务逻辑留在对应策略 service 中

当前不建议提前引入以下内容：

1. 策略注册中心
2. 通用策略工厂
3. 统一策略调度器
4. 过早的插件化扩展层

这些抽象在只有一个策略时都没有真实收益，只会增加理解成本。等第二个策略真正落地时，再根据重复度决定是否抽象，是更合适的时机。

## 8. OpenAPI 与前端类型影响

采用“每策略独立路由”后，OpenAPI 表达会更直接：

1. 每个策略都有自己的请求体 schema 和响应体 schema。
2. 不需要在当前阶段就引入复杂的判别联合说明。
3. `openapi-typescript` 可以直接为前端生成各策略接口对应的类型。

这对当前项目有三个直接好处：

1. 前端更容易理解当前只支持哪个策略。
2. 页面层不需要自行维护第二套近似 DTO。
3. 以后新增策略时，只是 OpenAPI 新增一路 schema，而不是修改当前 DCA 请求形状。

## 9. 当前阶段还需要特别注意的点

### 9.1 不要出现双重策略来源

既然策略由路由表达，请求体中就不能再保留 `strategy_id` 作为分发依据。

### 9.2 不要把 DCA 结果误当成通用结果

如果将来发现某个结果字段只对 DCA 成立，就不应该继续留在统一响应外壳里，而应移动到 DCA 子响应或未来的策略专属结果区块。

### 9.3 不要让 Base 退化成垃圾桶

后续新增策略时，最容易出现的坏味道不是字段不够，而是 Base 被持续塞入“看起来有点通用”的字段。只要一个字段的业务语义依赖具体策略，就不应进入 Base。

### 9.4 不要为了未来过早做策略平台

当前目标是打通单次回测闭环，不是做完整多策略执行平台。只要 DCA 能单独跑通、契约清晰、前端能消费类型，就已经满足当前主线。

## 10. 非目标

本次设计明确不处理以下内容：

1. 多策略统一调度入口
2. 策略注册中心
3. 通用策略插件系统
4. `strategy_result` 的提前占位实现
5. 真实回测引擎的数据源与撮合细节

## 11. 下一步建议

在本设计确认后，建议按以下顺序推进：

1. 调整单次回测后端契约，从通用 `/backtests/run` 切换到 `dca` 独立路由。
2. 将请求 schema 拆为 `Base + Dca`。
3. 将响应 schema 拆为 `Base + Dca`，并把 `input_snapshot` 下沉到 DCA 子响应。
4. 更新测试与 OpenAPI 文档。
5. 再进入 `openapi-typescript` 生成与前端类型消费。

## 12. 结论

对于当前项目，最合适的方案不是“为了平台化而统一入口”，而是：

1. 路由层按策略拆分。
2. 请求层用 `Base + 策略子请求`。
3. 响应层用统一外壳 + 策略子响应。
4. 请求不保留 `strategy_id`，响应保留 `strategy_id`。

这套方案在当前阶段更简单、更直观、更符合学习场景，同时没有破坏后续扩展路径。它保留了清晰的演进空间，但不提前承担不必要的抽象成本。
