# openapi-typescript 接入实施方案

## 1. 文档目的

本文用于把已经确认的方向落成可执行实施方案：

1. 后端继续使用 FastAPI + Pydantic 维护契约。
2. FastAPI 自动导出 OpenAPI。
3. 前端使用 `openapi-typescript` 只生成 TypeScript 类型。
4. 前端请求层继续手写，不生成完整 SDK 或 React Query hooks。

本文只定义落地方式，不直接引入依赖或修改源码。

## 2. 当前决策前提

当前项目已经确认：

1. OpenAPI 作为前后端契约的标准导出格式。
2. 当前项目采用 `openapi-typescript`。
3. 只生成 TypeScript 类型，不生成完整 client。
4. 后端 schema 是事实源，前端不再手写第二套核心接口类型。

## 3. 实施目标

接入完成后，应达到以下效果：

1. 后端一旦变更 schema，前端可以重新生成最新类型。
2. 前端请求函数直接引用生成出来的类型。
3. 前后端不再长期维护两套近似接口结构。
4. 整体流程足够轻，不引入过多生成物和额外抽象。

## 4. 推荐目录约定

### 4.1 OpenAPI 输入来源

后端直接提供：

`http://localhost:8000/api/v1/openapi.json`

说明：

1. 这是 FastAPI 自动导出的 OpenAPI 文档。
2. 不额外维护手写 YAML / JSON 契约文件。
3. 后端 schema 变更后，OpenAPI 自动反映最新结构。

### 4.2 前端生成类型文件位置

推荐生成到：

`frontend/src/types/api-schema.ts`

原因：

1. 路径简单直观。
2. 与“前端类型定义”职责一致。
3. 当前项目体量不大，没有必要提前拆更复杂的生成目录。

说明：

1. 这里放的是生成文件，不是手写文件。
2. 生成文件应作为前端接口类型唯一入口之一使用。

## 5. 依赖与命令策略

### 5.1 依赖策略

新增依赖仅需：

1. `openapi-typescript`

说明：

1. 当前不接入 Orval。
2. 当前不接入完整 client 生成器。
3. 当前不接入 mock 生成工具。

### 5.2 推荐执行命令

在仓库根目录执行的推荐命令形态：

```bash
pnpm add -D openapi-typescript --filter frontend
pnpm --filter frontend exec openapi-typescript http://localhost:8000/api/v1/openapi.json -o src/types/api-schema.ts
```

说明：

1. 第一条命令用于安装类型生成器。
2. 第二条命令用于从后端 OpenAPI 直接生成类型文件。
3. 由于当前协作边界规定依赖安装由你执行，所以这里只记录命令，不由助手代执行。

## 6. 前端消费方式

### 6.1 推荐用法

前端继续手写请求函数，例如放在：

`frontend/src/services/`

请求函数的入参、出参类型直接引用生成文件中的类型。

示意：

1. 请求体类型引用 `paths[...]` 或转换后的局部类型别名。
2. 响应体类型同样引用生成结构。

### 6.2 不推荐用法

当前不推荐：

1. 在组件里直接拼 URL 和手写类型。
2. 在组件里重复定义接口返回结构。
3. 一边保留手写接口类型，一边又生成同名近似类型。

## 7. 什么时候重新生成类型

建议在以下场景重新生成：

1. 后端 schema 字段有新增、删除、改名。
2. 后端响应结构有变化。
3. 错误响应结构有变化。
4. 新增接口进入前端消费范围。

不建议每次无关改动都重生成。

## 8. 推荐工作流

建议工作流如下：

1. 先修改后端 schema。
2. 启动后端服务，确认 OpenAPI 可访问。
3. 执行 `openapi-typescript` 生成前端类型。
4. 前端请求函数改用最新类型。
5. 前端页面再继续开发。

这条顺序的意义在于：

1. 后端契约始终是源头。
2. 前端不会先手写一份临时类型再回头删除。
3. 能最大程度减少字段漂移。

## 9. 可能遇到的问题

### 9.1 后端未启动导致无法生成

表现：

1. 命令无法访问 `http://localhost:8000/api/v1/openapi.json`

处理：

1. 先确认后端已启动。
2. 先确认 `/api/v1/docs` 可打开。

### 9.2 生成文件过大、可读性一般

表现：

1. `api-schema.ts` 会包含完整 OpenAPI 类型结构。

处理：

1. 这是当前方案的可接受代价。
2. 前端业务代码不直接在组件里操作整棵类型树，而是在服务层做局部类型别名。

### 9.3 想要 mock 或完整 client

说明：

1. 当前方案不阻塞未来升级。
2. 如果后续确实需要 mock、自动 client、React Query hooks，可以在保持 OpenAPI 不变的前提下再评估 Orval。
3. 当前不要为了未来 mock 提前切换工具。

## 10. 当前阶段不做的事

1. 不生成完整 SDK。
2. 不生成请求函数。
3. 不生成 React Query hooks。
4. 不生成 mock。
5. 不引入多套 OpenAPI 消费工具并存。

## 11. 最终实施建议

当前项目的推荐实施方式是：

1. 后端保持 FastAPI/Pydantic schema。
2. 前端新增 `openapi-typescript` 一个工具依赖。
3. 生成类型文件到 `frontend/src/types/api-schema.ts`。
4. 前端服务层引用生成类型，组件层不再手写接口结构。

这条路线既满足当前项目“单一契约源”的目标，也不会把项目一下子推到复杂生成体系里。
