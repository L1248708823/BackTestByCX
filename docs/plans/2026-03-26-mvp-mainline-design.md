# MVP 主线收敛设计文档

## 1. 文档目的

本文用于回答三个直接影响推进的问题：

1. 这个项目当前真正要做什么。
2. 这个项目现在做到哪一步了。
3. 后续应该按什么顺序推进，才能在最短路径内完成 MVP。

本文不是 PRD 的替代品，而是对当前执行主线的重新收敛，目标是减少认知混乱和无效分支。

## 2. 当前项目要做什么

项目目标不是做一个通用量化平台，也不是先做一个复杂炫目的展示站。

当前 MVP 的唯一目标是交付一个本地可运行的个人 A 股 ETF 回测工具，满足以下能力：

1. 用户可以配置参数并发起一次单次回测。
2. 系统可以返回可信、可复核的结构化回测结果。
3. 用户可以执行随机实验并查看统计分布。
4. 用户可以基于内部结果生成一份结构稳定的 AI 报告。

项目边界保持收缩：

1. 资产范围仅 A 股 ETF。
2. 策略范围仅定投策略 `dca`。
3. AI 只做结果解释，不参与策略搜索、自动调参或联网分析。
4. 首版只追求可运行、可验证、可演示，不追求平台化和高度抽象。

## 3. 当前做到哪一步了

### 3.1 已完成

1. 已有 PRD，产品边界基本明确。
2. 已有技术设计文档，技术栈和分层方向已明确。
3. 前端已有 React 基础骨架、路由和页面占位。
4. 后端已有 FastAPI 最小骨架、基础 schema、service 和健康检查测试。
5. 已有安装、启动、人工操作台账等配套文档。

### 3.2 未完成但会误导认知的点

1. 工程验证闭环还没有完全确认。
2. 回测接口、实验接口、报告接口的字段契约还没有冻结。
3. 前端三页仍以占位为主，没有业务闭环。
4. 后端还没有完成数据源接入、定投策略实现和随机实验实现。

结论：当前状态应定义为“骨架阶段已完成，MVP 业务闭环尚未完成”。

## 4. 主线判断

当前真正的执行主线只有一条：

先完成工程可运行验证，再冻结契约，再打通单次回测闭环，随后扩展随机实验和 AI 报告，最后补测试与演示材料。

这条主线背后的原因如下：

1. 不先验证工程基线，后续开发容易建立在不稳定环境上。
2. 不先冻结契约，前后端会在字段命名和结构上持续返工。
3. 不先打通单次回测，随机实验和 AI 报告都没有可靠输入。
4. 不先收缩 MVP 范围，项目会被展示层、动效和过度抽象拖慢。

## 5. MVP 完成清单

### 5.1 P0：基线闭环

目标：确认工程不是“写了骨架但不能稳定运行”。

需要完成：

1. 前端依赖与构建验证通过。
2. Turbo 任务编排验证通过。
3. 后端 Python 3.12 环境验证通过。
4. 后端服务启动、健康检查、OpenAPI、测试验证通过。
5. `localDoc/TODO.md` 与当前项目状态同步。

完成标准：

1. 前端能启动并完成构建。
2. 后端能启动并通过现有测试。
3. 当前阻塞项有明确日志和结论，不再停留在“可能有问题”。

### 5.2 P1：契约冻结

目标：消除字段口径漂移，建立唯一事实来源。

需要完成：

1. 定义单次回测请求结构。
2. 定义单次回测结果结构。
3. 定义随机实验请求结构。
4. 定义随机实验结果结构。
5. 定义 AI 报告请求与响应结构。
6. 定义统一错误响应结构。
7. 定义策略接口协议和版本字段。

完成标准：

1. 前后端都以同一份契约为准。
2. 不允许页面层、组件层、hook 层做字段兼容兜底。
3. 回测结果结构可以同时服务前端展示和 AI 报告。

### 5.3 P2：单次回测闭环

目标：先把最核心的一条业务链打通。

需要完成：

1. 接入一个 A 股 ETF 数据源，优先 `AkShare`。
2. 完成最小行情标准化。
3. 实现定投策略 `dca`。
4. 实现最小成本模型。
5. 实现关键指标计算。
6. 提供 `/api/backtests/run`。
7. 前端参数页可提交请求。
8. 前端结果页可展示指标、曲线和明细。

完成标准：

1. 用户可以输入参数并成功完成一次回测。
2. 结果包含指标、曲线数据、输入快照、版本信息。
3. 同参数同数据重复运行结果一致。

### 5.4 P3：随机实验闭环

目标：补上项目的第二个核心价值点，即鲁棒性观察。

需要完成：

1. 实现随机起点 + 随机持有期实验逻辑。
2. 支持实验次数、最短持有天数、最长持有天数、随机种子。
3. 输出收益分布、回撤分布、分位数统计。
4. 提供 `/api/experiments/run`。
5. 前端实验页可提交请求并展示统计结果。

完成标准：

1. 固定 `random_seed` 时结果一致。
2. 用户可以把单次回测结果放进统计分布中观察位置。

### 5.5 P4：AI 报告闭环

目标：把结构化结果转换成可展示、可复盘的自然语言材料。

需要完成：

1. 定义报告输入载荷。
2. 实现 Prompt 模板。
3. 实现 Provider 适配层。
4. 支持主模型和降级模型。
5. 提供 `/api/reports/generate`。
6. 前端可查看报告内容和生成状态。

完成标准：

1. AI 只读取内部结构化结果。
2. 报告章节稳定，便于比较不同实验。
3. 模型失败时返回结构化错误，不吞异常。

### 5.6 P5：测试与演示收尾

目标：让项目从“能跑”提升到“可信、可演示”。

需要完成：

1. 单元测试覆盖成本计算、指标计算、采样边界。
2. 集成测试覆盖回测、实验、报告主链路。
3. 至少准备一组演示参数。
4. 补齐最小导出能力，首版可只做 JSON。
5. 更新文档，确保按手册可以复现。

完成标准：

1. 关键逻辑有最小测试护栏。
2. 演示流程可以被稳定复现。

## 6. 非 MVP 范围

以下事项明确不进入当前主线：

1. 个股支持。
2. 多策略并行支持。
3. 自定义策略 DSL。
4. 复杂首页动效或展示层重构。
5. 为未来扩展而提前引入的重型抽象。
6. 复杂数据库设计和过早的平台化能力。

## 7. 推荐执行顺序

建议严格按下面顺序推进：

1. 工程验证闭环。
2. 契约冻结。
3. 单次回测闭环。
4. 随机实验闭环。
5. AI 报告闭环。
6. 测试、导出、演示收尾。

不建议改变顺序的原因：

1. 单次回测是随机实验和 AI 报告的输入基础。
2. 契约没定时，前后端同时开发会产生高返工成本。
3. 过早优化界面表现对 MVP 交付价值很低。

## 8. 风险提醒

1. 当前最大的风险不是不会写，而是主线漂移。
2. 当前第二大的风险是字段口径不统一，最终逼出一堆兼容兜底代码。
3. 当前第三大的风险是为了“看起来更完整”而提前做超出 MVP 的设计。

针对以上风险，当前阶段应坚持三个原则：

1. 先完成闭环，再谈扩展。
2. 先统一契约，再写页面。
3. 先保证结果可信，再优化展示效果。

## 9. 本文结论

当前项目的正确推进方式不是继续扩散需求，也不是继续修饰展示层，而是围绕一条清晰主线推进：

先验证工程基线，再冻结契约，再完成单次回测、随机实验和 AI 报告三段式闭环，最后补测试和演示材料。

只有做到这一步，项目才算真正进入“可交付的 MVP”状态。
