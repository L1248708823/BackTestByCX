# OpenAPI 与 TypeScript 类型产出方案对比

## 1. 文档目的

本文用于回答两个问题：

1. 当前项目是否应把 OpenAPI 作为前后端契约导出物。
2. 当前项目的 TypeScript 类型应如何产出，才能在保证单一契约源的同时避免过度设计。

本文只讨论方案，不直接引入新依赖或改造代码。

## 2. 当前项目背景

当前项目已经具备这些前提：

1. 后端使用 FastAPI + Pydantic，对外接口天然可导出 OpenAPI。
2. 前端使用 React + TypeScript，后续一定需要消费后端契约类型。
3. 项目当前原则已经明确：
   - 不允许兼容或兜底代码。
   - 前后端数据结构尽量只维护一套。
   - 避免过度设计。
   - 只做当前直接需要或已明确约定的扩展点。

因此，这里真正要解决的问题不是“能不能生成类型”，而是“如何让前后端围绕同一份契约协作，同时不要把工具链搞得太重”。

## 3. 市面上主流方案

### 方案 A：手写前端 TypeScript 类型

做法：

1. 后端继续维护 Pydantic schema。
2. 前端手写一份对应的 TypeScript 类型。
3. 请求函数和类型都由前端自行维护。

优点：

1. 不需要新增代码生成工具。
2. 上手成本最低。
3. 适合非常小、非常稳定、接口几乎不变的项目。

缺点：

1. 前后端天然存在“双份结构”。
2. 字段一变就容易漂移。
3. 与当前项目“前后端尽量只维护一套结构”的原则冲突。

适用场景：

1. 单文件 demo。
2. 接口极少且几乎不变化的临时项目。

结论：

当前项目不推荐继续长期使用该方案。

### 方案 B：FastAPI 导出 OpenAPI，只生成 TypeScript 类型

做法：

1. 后端 Pydantic schema 作为事实源。
2. FastAPI 自动导出 OpenAPI JSON。
3. 前端通过生成器只生成 TypeScript 类型定义。
4. 前端请求函数继续手写，保持最小可控。

优点：

1. 后端 schema 是唯一事实源，契约不容易漂移。
2. 前端不需要再维护第二套手写类型。
3. 工具链相对轻量，复杂度可控。
4. 与当前项目“单一数据结构口径 + 避免过度设计”的原则最匹配。

缺点：

1. 需要增加一个生成步骤。
2. 需要约定何时重新生成类型。

适用场景：

1. 前后端分离但同仓库的中小项目。
2. 已有明确后端 schema，希望前端少写重复类型的项目。

结论：

这是当前项目最平衡的方案。

### 方案 C：基于 OpenAPI 生成完整前端 Client / SDK

做法：

1. 后端导出 OpenAPI。
2. 使用代码生成器产出：
   - TypeScript 类型
   - 请求函数
   - 甚至 Query Hooks / Mock / Zod Schema

优点：

1. 契约统一程度最高。
2. 请求层、类型层、部分状态层都可自动生成。
3. 对大团队、多接口项目的收益很高。

缺点：

1. 生成物更多，学习成本更高。
2. 项目初期容易把请求层也交给生成器，降低对实际代码流向的掌控。
3. 一旦配置不当，容易引入额外结构、额外目录和额外抽象。

适用场景：

1. 接口数量较多。
2. 团队多人并行。
3. 已经明确接受代码生成主导 API 消费层。

结论：

当前项目阶段不推荐直接上这一档。

### 方案 D：使用跨语言通用代码生成器生成 SDK

代表工具：

1. OpenAPI Generator
2. Swagger Codegen

做法：

1. 使用 OpenAPI 规范输入。
2. 生成多语言客户端或服务端代码。

优点：

1. 工具成熟，生态广。
2. 适合多语言、多端、多团队协作。
3. 若未来需要生成 Python/Java/TS 等多个客户端，有统一入口。

缺点：

1. 对当前项目来说太重。
2. 生成产物通常偏多，定制成本更高。
3. 当前只有前端 TS 类型需求，用它们会显得明显过度。

适用场景：

1. 多语言 SDK 分发。
2. 企业级 API 平台。

结论：

当前项目不推荐。

## 4. 主流工具代表

### 4.1 openapi-typescript

定位：

1. 专注把 OpenAPI schema 转成 TypeScript 类型。
2. 不强制生成请求客户端。

特点：

1. 轻量。
2. 运行时零依赖风格更明显。
3. 非常适合“只要类型，不要 SDK”。

适合当前项目的点：

1. 与 FastAPI 导出的 OpenAPI 能自然衔接。
2. 不会逼你把请求层也生成掉。
3. 更符合当前“最小够用”的原则。

### 4.2 Orval

定位：

1. 基于 OpenAPI 生成 TypeScript 类型、请求函数、React Query Hook、Mock 等。

特点：

1. 面向前端体验非常完整。
2. 对 React Query、SWR、MSW 等集成很好。
3. 更像“前端 API 消费层全家桶生成器”。

适合当前项目的点：

1. 如果以后接口越来越多，前端请求层样板代码会明显减少。

当前阶段的问题：

1. 对当前项目来说略重。
2. 现在上它，容易把“类型生成”一步升级成“整套请求层生成”。

### 4.3 Hey API / openapi-ts

定位：

1. 从 OpenAPI 生成 SDK、类型、插件扩展等。

特点：

1. 更现代，插件体系更强。
2. 能覆盖更完整的前端代码生成场景。

当前阶段的问题：

1. 和 Orval 类似，能力强，但当前项目未必需要那么多层。
2. 更适合已经决定“代码生成主导 API 层”的团队。

### 4.4 OpenAPI Generator / Swagger Codegen

定位：

1. 通用型、多语言代码生成器。

特点：

1. 非常成熟，覆盖多语言、多客户端场景。
2. 更偏“平台型代码生成工具”。

当前阶段的问题：

1. 对当前单仓库前端项目过重。
2. 不是最贴合“只为前端产出 TS 类型”的工具。

## 5. 我的推荐方案

### 推荐结论

当前项目推荐采用：

**FastAPI/Pydantic 作为契约源 + OpenAPI 作为契约导出格式 + `openapi-typescript` 只生成前端 TypeScript 类型**

不推荐当前就生成完整 SDK / Client / React Query Hook。

### 为什么这是当前最好的方案

原因有五个：

1. 它最符合“前后端只维护一套结构”的原则。
2. 它不会把当前项目一下子推到复杂代码生成体系里。
3. 它保留了前端请求层的显式代码，可读性更高，学习成本更低。
4. 它对当前接口数量和项目规模来说刚刚好。
5. 它给未来保留了升级空间：后续如果接口明显变多，再升级到 Orval 或 Hey API 也不迟。

## 6. 不推荐当前直接上 Orval / 全量 Client 生成的原因

不是这些工具不好，而是当前阶段不合适。

核心原因：

1. 你现在还在学习阶段，需要先把契约和请求流向看清楚。
2. 当前接口数量很少，生成完整 client 的收益还不够高。
3. 当前项目规则强调“避免过度设计”，而全量 client 生成更像第二阶段优化，不是第一阶段基础建设。

换句话说：

当前阶段真正急需自动化的是“类型同步”，不是“请求层自动化”。

## 7. 当前项目的推荐落地方式

建议分两步走：

### 第一步：把 OpenAPI 当成唯一契约导出物

做法：

1. 后端继续维护 Pydantic schema。
2. FastAPI 自动暴露 OpenAPI。
3. 前端不再手写第二套核心接口类型。

### 第二步：只生成前端类型文件

做法：

1. 生成一份集中式类型文件。
2. 前端请求函数继续自己写。
3. 请求函数的入参和出参直接引用生成出来的类型。

## 8. 当前不建议做的事

1. 不建议一上来生成完整 SDK。
2. 不建议一上来生成 React Query hooks。
3. 不建议为 OpenAPI 引入多层包装配置。
4. 不建议同时保留“手写 TS 类型 + 生成 TS 类型”双轨结构。

## 9. 最终结论

从当前项目阶段、学习目标和规则约束综合来看：

1. OpenAPI 应该保留，并作为前后端契约的标准导出格式。
2. TypeScript 类型应该自动生成，而不是继续手写维护。
3. 当前最佳方案不是全量代码生成，而是“只生成类型”。
4. 在候选工具中，当前阶段最推荐 `openapi-typescript`。
